2

Las reglas del equipo

Steering ▸
La IA no sabe nada de tu equipo. Hasta que le dices.

El problema

Cada dev le pide cosas distintas a la IA. El codigo resultante es inconsistente. Nombres en ingles, errores en ingles, sin documentacion. Un caos.

La solucion — Steering

Escribe en el chat:

Prompt Crea un steering file con las convenciones de nuestro proyecto: - Los mensajes de error deben estar en espanol - Usar camelCase para variables y funciones - Todas las funciones deben tener comentarios JSDoc - Siempre manejar errores con try/catch - Separa el codigo en html, css, y js - El color principal de la app es #FF9900 y #000000 - La tipografia debe ser Roboto como primera opcion

Kiro crea .kiro/steering/convenciones.md. Abrelo y muestralo.

Ahora pruebalo

Prompt Refactoriza el conversor de temperaturas siguiendo las convenciones del proyecto.

Senala en el diff: JSDoc en cada funcion, errores en espanol, try/catch donde antes no habia nada.

Lo que dices

"No le repeti las reglas. Las leyo solas. Esto aplica en cada prompt, para siempre, para todo el equipo."

3

Una feature de verdad, con plan

Specs ▸
El cliente quiere historial de conversiones. Hagamoslo bien.

El problema

Un prompt largo para una feature compleja suele salir mal. La IA adivina lo que quieres. A veces acierta, a veces no.

La solucion — Specs

En Kiro

Panel lateral → Specs → boton "+" → Nombre: historial-conversiones

Descripcion Agregar historial de conversiones al conversor. El usuario ve sus ultimas 10 conversiones, puede limpiar el historial y hacer clic en una entrada para reutilizarla. Los datos persisten con localStorage.

Kiro genera 3 documentos: Requerimientos (que debe hacer), Diseno (como lo va a implementar), Tareas (pasos concretos de codigo). Aprueba cada seccion despacio. Que se vea que tu decides.

Lo que dices

"No es un prompt que espero que funcione. Es un plan que yo revise y aprobe. Kiro lo ejecuta paso a paso."

4

El IDE que trabaja solo

Hooks ▸
¿Y si el IDE pudiera vigilar el codigo por ti?

El problema

Guardas un archivo. Hay un error. Te enteras en produccion.

La solucion — Hooks

En Kiro

Panel lateral → Agent Hooks → "+"

Prompt Cuando guarde un archivo JavaScript, revisa si hay errores o funciones sin documentacion JSDoc y avisame en el chat.

Demuestralo en vivo

  1. Abre app.js
  2. Borra un parentesis de cierre en cualquier funcion
  3. Guarda con Cmd+S
  4. Mira el chat
Lo que dices

"No corri ningun comando. No instale ningun linter. El IDE reacciono solo al guardar."

Segundo demo (mismo hook, otro caso)

Crea utils.js con una funcion sin documentar:

function sumar(a, b) { return a + b; }

Guarda con Cmd+S. El mismo hook detecta que falta JSDoc y lo agrega automaticamente.

Lo que dices

"Un solo hook. Detecta errores y documenta funciones. No necesite dos reglas separadas."

5

Agentes con criterio propio

Agents ▸
¿Y si pudieras tener un senior dev disponible 24/7?

El problema

La IA generica aprueba cualquier cosa. No tiene criterio de tu equipo.

La solucion — Custom Agents

Prompt Crea un agente llamado "revisor" que actue como un senior developer estricto con la calidad. Solo aprueba codigo que tenga manejo de errores explicito y funciones documentadas con JSDoc.

Luego invocalo:

Prompt @revisor revisa app.js y dime si lo aprobarias para produccion.
Lo que dices

"No es un asistente generico. Tiene el criterio de tu equipo. Puedes tener uno para seguridad, otro para accesibilidad, otro para performance."

6

Expertise bajo demanda

Skills ▸
Steering siempre esta activo. Skills se activan cuando las necesitas.

El problema

El steering es perfecto para reglas del equipo, pero no quieres cargar conocimiento especializado en cada prompt. Un contexto lleno de cosas irrelevantes es contexto desperdiciado.

La solucion — Skills

Prompt Crea una skill llamada "fisica-temperaturas" que sepa sobre escalas de temperatura, sus rangos fisicos validos, y casos extremos como el cero absoluto.

Kiro crea .kiro/skills/fisica-temperaturas/SKILL.md.

Demuestralo

Prompt ¿Que pasa si el usuario intenta convertir -300°C?

Kiro detecta que la pregunta es sobre fisica de temperaturas, activa la skill automaticamente y responde con criterio: el cero absoluto es -273.15°C, ese valor es fisicamente imposible.

Lo que dices

"No cargue nada manualmente. Kiro supo cuando necesitaba ese conocimiento. Steering son las reglas. Skills es la expertise. Cada una carga cuando corresponde."

7

La IA que busca sola

Web Search ▸
¿Para que cambiar de pestana si Kiro puede buscar por ti?

El problema

Estas en el IDE y necesitas un dato puntual: una temperatura de referencia, un estandar, una constante fisica. Sales al browser, buscas, volves. Rompe el flow.

La solucion — Web Search integrado

Sin instalar nada, escribe directamente en el chat:

Prompt ¿Cual fue la temperatura mas baja registrada en la Tierra y donde?

Kiro busca en internet en tiempo real y responde con el dato y la fuente.

Segundo ejemplo

Prompt ¿Cual es el estandar ISO para representar temperaturas en interfaces de usuario?
Lo que dices

"No abri ninguna pestana. No copie nada. La informacion llego al contexto donde la necesitaba."

8

Conectado al ecosistema AWS

MCP ▸
Kiro no vive en una burbuja. Se conecta a tus herramientas.

El problema

Queres hacer deploy de la app. La IA generica te da respuestas genericas. No sabe que servicios de AWS existen hoy, ni como configurarlos correctamente.

La solucion — MCP (Model Context Protocol)

En Kiro

Abre .kiro/settings/mcp.json y agrega:

{ "mcpServers": { "awslabs.aws-documentation-mcp-server": { "command": "uvx", "args": ["awslabs.aws-documentation-mcp-server@latest"], "env": { "FASTMCP_LOG_LEVEL": "ERROR" }, "disabled": false, "autoApprove": [] } } }

El servidor se conecta automaticamente. Ahora pregunta:

Prompt ¿Cual es la mejor forma de hacer deploy de esta app a un servicio de hosting estatico en AWS? Busca en la documentacion oficial.

Kiro consulta la documentacion real de AWS en tiempo real — S3 + CloudFront, configuracion de bucket, politicas — y te da una respuesta basada en docs actualizados, no en su entrenamiento.

Muestra en el chat como cita las fuentes de la documentacion oficial.

Lo que dices

"No es una respuesta generica. Es la documentacion oficial de AWS, leida en tiempo real, aplicada a tu proyecto especifico. Eso es lo que hace MCP: conecta la IA con tus herramientas reales."

9

Todo empaquetado: Powers

Powers ▸
MCP es el protocolo. Powers es la experiencia lista para usar.

El problema

Todo lo que construimos (steering, hooks, skills, agents, MCP) funciona, pero vive disperso en .kiro/. Si otro dev empieza de cero, tiene que configurar todo a mano.

La solucion — Powers

Un Power empaqueta MCP servers + steering + hooks + instrucciones de onboarding en una sola unidad instalable.

Ejemplo real: Instalamos el Power de "Build full-stack apps with AWS Amplify" desde el catalogo oficial de Powers. Nos da el MCP server de Amplify + las instrucciones de como usarlo para deploy, auth, storage y mas.

Ejemplo avanzado: Usamos el Power "Build a Power" para crear nuestro propio Power que empaqueta todo lo que hicimos en esta demo: convenciones, skill de fisica, hook de revision, agente revisor y MCP de AWS.

Los Powers del catalogo oficial estan curados por partners (AWS, Stripe, Figma, etc.) y la comunidad. Tambien podes crear los tuyos propios y compartirlos via GitHub.

Lo que dices

"Con MCP conectaste la IA a una herramienta. Con Powers le diste el manual de como usarla. Un clic. Sin configuracion. Listo para el equipo."
