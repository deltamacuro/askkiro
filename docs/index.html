Fase 1 · Steering

Haz que Kiro siga TUS reglas

Cada dev le pide cosas distintas. El codigo es un caos.

Kiro Chat
Crea un steering file con las convenciones de nuestro proyecto: - Los mensajes de error deben estar en espanol - Usar camelCase para variables y funciones - Todas las funciones deben tener comentarios JSDoc - Siempre manejar errores con try/catch - Separa el codigo en html, css, y js - El color principal de la app es #FF9900 y #000000 - La tipografia debe ser Roboto como primera opcion

Ahora pruebalo:

Kiro Chat
Refactoriza el conversor de temperaturas siguiendo las convenciones del proyecto.

Resultado: JSDoc, try/catch, errores en espanol. Sin repetir las reglas.

Lo que dices

"No le repeti las reglas. Las leyo solas. Para siempre, para todo el equipo."

Fase 1 · Specs

Planifica antes de codear

El cliente quiere historial. Un prompt largo sale mal.

En Kiro Panel lateral → Specs → "+" → historial-conversiones
Descripcion
Agregar historial de conversiones al conversor. El usuario ve sus ultimas 10 conversiones, puede limpiar el historial y hacer clic en una entrada para reutilizarla. Los datos persisten con localStorage.

Resultado: requirements, design y tasks. Tu apruebas cada paso.

Lo que dices

"No es un prompt que espero que funcione. Es un plan que yo revise y aprobe."

Fase 2 · Hooks

El IDE que vigila por ti

Guardas un archivo. Hay un error. Te enteras en produccion.

En Kiro Panel lateral → Agent Hooks → "+"
Kiro Chat
Cuando guarde un archivo JavaScript, revisa si hay errores o funciones sin documentacion JSDoc y avisame en el chat.

Pruebalo: borra un parentesis en app.js, guarda con Cmd+S.

Resultado: Kiro detecta el error automaticamente.

Lo que dices

"No corri ningun comando. El IDE reacciono solo."

Fase 2 · Agents

Tu senior dev personal

La IA generica aprueba cualquier cosa. Dale criterio.

Kiro Chat
Crea un agente llamado "revisor" que actue como un senior developer estricto con la calidad. Solo aprueba codigo que tenga manejo de errores explicito y funciones documentadas con JSDoc.

Invocalo:

Kiro Chat
@revisor revisa app.js y dime si lo aprobarias para produccion.

Resultado: APROBADO o RECHAZADO con detalles.

Lo que dices

"Tiene el criterio de tu equipo."

Fase 2 · Skills

Dale expertise a Kiro

Steering son las reglas. Skills es el conocimiento.

Kiro Chat
Crea una skill llamada "fisica-temperaturas" que sepa sobre escalas de temperatura, sus rangos fisicos validos, y casos extremos como el cero absoluto.

Pruebalo:

Kiro Chat
¿Que pasa si el usuario intenta convertir -300°C?

Resultado: Kiro sabe que -300°C esta por debajo del cero absoluto.

Lo que dices

"Kiro supo cuando necesitaba ese conocimiento."

Fase 3 · MCP

Conecta con AWS en tiempo real

La IA generica no sabe que servicios de AWS existen hoy.

En Kiro Configura .kiro/settings/mcp.json con el servidor de docs de AWS.
Kiro Chat
¿Cual es la mejor forma de hacer deploy de esta app a un servicio de hosting estatico en AWS? Busca en la documentacion oficial.

Resultado: respuesta basada en docs oficiales de AWS.

Lo que dices

"Documentacion oficial, leida en tiempo real."

Fase 3 · Powers

Empaqueta todo en un clic

Todo funciona pero vive disperso. Empaquetalo.

Un Power empaqueta MCP + steering + hooks + onboarding. Usa "Build a Power" para crear el tuyo.

Resultado: un paquete que cualquier dev instala y tiene todo.

Lo que dices

"Con MCP conectaste la IA. Con Powers le diste el manual. Un clic."
